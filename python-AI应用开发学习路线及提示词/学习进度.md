### 我真正理解了什么

- 多轮对话的本质不是模型“记住了我”，而是程序把之前的问题和回答重新组织成上下文发给模型
- 如果不传历史，模型看到的就是一次全新的独立请求
- 流式输出不是换了一个新接口，而是在同一个调用方法上改变返回方式
- `stream=True` 之后，返回值不再是一次性完整文本，而是可迭代的数据块，所以必须循环读取
- 流式输出除了提升体验，还有一个工程上的意义：后续可以边返回前端边收集完整文本落库
- 流式模式下读取的不是最终完整回答，而是每次新增的增量内容，所以字段从 `message.content` 变成了 `delta.content`
- 流式输出时，后端常见做法是一边把增量内容实时推给前端，一边在内存中拼接 `full_text`，等流结束后再一次性持久化完整回答

### 还不理解的点

- 多轮对话历史如果特别长，如何裁剪上下文、降低 token 成本，还需要后续学习
- 多轮对话历史如何持久化到数据库、如何在真实服务中恢复会话，还需要结合 FastAPI 项目继续学习

### 我真正理解了什么

- FastAPI 的执行链路可以理解为：请求进入服务 -> 根据路径匹配 router -> 校验请求体 -> 调用 service -> 得到结构化结果 -> 转成响应模型 -> 序列化为 JSON 返回
- `APIRouter` 的价值不是“多写几个文件”，而是把不同业务接口拆分到不同模块中，类似 Java 中按 Controller 拆分接口
- `include_router(...)` 的作用是把各个 router 注册到主应用 `app` 中，否则这些接口不会真正暴露出去
- `prefix="/api/chat"` 配合 `@router.post("")` 可以组成最终接口路径 `/api/chat`
- 请求体和响应体单独放在 `schemas` 中，本质上类似 Java 项目里的 DTO / VO，可以让接口结构更清晰
- 业务调用放在 `services` 中，是为了避免接口层塞满业务逻辑，后续更容易维护、测试和复用
- `response_model=ChatResponse` 不只是给文档看的，它还会约束返回结构、参与序列化，并让接口输出更稳定
- `.env -> settings.py -> service/router` 是比较常见的本地开发配置链路，生产环境可以再升级为 Docker 环境变量、配置中心或 Secret 管理

### 还不理解的点

### 我真正理解了什么

- FastAPI 不是靠 `data: ...\n\n` 识别 SSE，而是通过 `StreamingResponse` 持续输出，通过 `media_type="text/event-stream"` 告诉客户端按 SSE 处理
- `data: ...\n\n` 是 SSE 每条消息的格式，`\n\n` 表示一条消息结束

- 流式接口开始返回后，如果中途异常，不能再返回普通 `JSONResponse`，应该继续用 SSE 协议发 `error` 事件
- 前端收到 `delta` 时追加显示，收到 `done` 时结束 loading，不应该再把完整回答追加一次
- 后端在流式过程中同步拼接完整回答，是为了后续保存会话历史、审计、上下文恢复和问题排查
