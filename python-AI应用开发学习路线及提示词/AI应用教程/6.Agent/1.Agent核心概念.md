## 一、Agent 是什么

一句话：Agent 是一个能自主完成任务的 AI 程序。

你之前学的所有东西，都是 Agent 的零件：

```
Skills 阶段：学会了调用 AI API（引擎）
Ollama 阶段：学会了本地部署模型（备用引擎）
MCP 阶段：学会了连接外部工具（接口标准）
RAG 阶段：学会了让 AI 基于文档回答（知识库）
Agent Skills 阶段：学会了工具调用、记忆、规划（核心能力）

Agent 阶段：把所有零件组装成一辆完整的车
```

## 二、Agent vs 普通 AI 应用

### 普通 AI 应用（你之前写的）

```
用户提问 → 调用 API → 返回回答 → 结束
```

特点：一问一答，你写死了流程，AI 只负责生成文字。

### Agent

```
用户给一个任务 → AI 自己思考该怎么做 → 自己选工具 → 自己执行 → 
看结果 → 不满意就再来 → 满意了才返回最终结果
```

特点：AI 自己决定做什么、怎么做、什么时候停。

### 关键区别

| | 普通 AI 应用 | Agent |
|--|--|--|
| 谁决定流程 | 你（开发者）写死 | AI 自己决定 |
| 能做几步 | 通常 1 步 | 多步，自己决定几步 |
| 能用工具吗 | 不能或固定调用 | 自己选择用哪个工具 |
| 能纠错吗 | 不能 | 能，发现错误可以重试 |
| 什么时候停 | 代码写死 | AI 自己判断任务完成 |

## 三、Agent 的完整架构

Agent 的工作就是一个循环：感知 → 思考 → 行动 → 观察

```
         ┌─────────────────────────────┐
         │                             │
         ▼                             │
    ┌─────────┐                        │
    │  感知    │ ← 接收用户输入/工具结果  │
    │ Perceive│                        │
    └────┬────┘                        │
         │                             │
         ▼                             │
    ┌─────────┐                        │
    │  思考    │ ← LLM 推理：该做什么？  │
    │  Think  │   用哪个工具？           │
    └────┬────┘                        │
         │                             │
         ▼                             │
    ┌─────────┐                        │
    │  行动    │ ← 调用工具/执行操作     │
    │   Act   │                        │
    └────┬────┘                        │
         │                             │
         ▼                             │
    ┌─────────┐    任务没完成            │
    │  观察    │ ──────────────────────→│
    │ Observe │    继续循环              │
    └────┬────┘                        
         │                             
         │ 任务完成                     
         ▼                             
    ┌─────────┐                        
    │  输出    │ → 返回最终结果给用户     
    │ Output  │                        
    └─────────┘                        
```

对应到代码，就是你在 Agent Skills Demo 2 里写的 for 循环：


注意区分下面的过程调用了2次API
response = client.chat.completions.create(...) 调用AI API
execute_tools(...) 调用工具函数或者工具API

```python
for iteration in range(max_iterations):
    # 循环最多跑 max_iterations 次（比如 5 次），这是安全阀，防止 AI 一直调工具停不下来

    response = client.chat.completions.create(...)
    # 调用 AI API（比如 DeepSeek API），把 messages 发给 AI
    # AI 在这一次调用中完成：看 messages 所有内容 → 思考分析 → 做出决策
    # 返回的 response 里包含 AI 的决策：要调工具（tool_calls 有值）还是直接回答（tool_calls 为 None）

    if response.tool_calls:
        # AI 决定要调用工具（比如 AI 说"我要调 get_weather('北京')"）

        execute_tools(...)
        # 执行工具函数（demo 里是本地模拟函数，实际项目中可能是调用真实的第三方 API）
        # 注意：这里调的是你自己写的工具函数，不是 AI API

        # 把工具执行结果以 {"role": "tool", "content": "晴天，35°C"} 加入 messages
        # 这样下一轮循环调用 AI API 时，AI 就能"看到"这个结果

        # 继续循环：回到 for 开头，再次调用 AI API
        # AI 看到 messages 里的工具结果，再次做出判断：还需要调工具 or 信息够了

    else:
        return response.content
        # AI 没有返回 tool_calls，说明信息够了，直接生成了文字回答
        # return 跳出整个函数，把回答返回给用户，循环结束
```

用 for + max_iterations 比 while True 更安全，能防止无限循环。

**逐行讲解：**

```python
for iteration in range(max_iterations):
```
循环最多跑 max_iterations 次（比如 5 次）。这是安全阀，防止 AI 一直调工具停不下来。

```python
    response = client.chat.completions.create(...)  # 思考
```
调用 AI API，把当前的 messages（包含用户问题 + 之前所有工具结果）发给 AI。AI 看完所有信息后做出判断：我是直接回答，还是需要调用工具？这一步就是 Agent 的"大脑在思考"。

```python

```
检查 AI 的返回：tool_calls 有值说明 AI 决定调用工具，tool_calls 为 None 说明 AI 觉得信息够了，可以直接回答。

```python

```
AI 说"我要调 get_weather("北京")"，你的代码就真的去执行 get_weather("北京")，拿到结果 "晴天，35°C"。

```python
        # 结果加入 messages  # 观察
```
把工具执行结果以 `{"role": "tool", "content": "晴天，35°C"}` 的形式追加到 messages 列表里。这样下一轮循环调用 API 时，AI 就能"看到"这个结果。

```python
        # 继续循环
```
回到 for 循环的开头，再次调用 API。这次 AI 看到了工具结果，可能决定：还需要调另一个工具（继续循环），或者信息够了（走 else 分支）。

```python

        return response.content  # 任务完成，输出
```
AI 没有返回 tool_calls，说明它觉得信息够了，直接生成了文字回答。return 跳出整个函数，把回答返回给用户。循环结束。

**用一个具体例子串起来：**

```
用户问："北京和上海哪个更热？"

第 1 轮循环：
  AI 思考 → 我需要查北京天气 → tool_calls = get_weather("北京")
  你执行 → 结果 "35°C" 加入 messages
  继续循环

第 2 轮循环：
  AI 看到北京 35°C → 还需要查上海 → tool_calls = get_weather("上海")
  你执行 → 结果 "22°C" 加入 messages
  继续循环

第 3 轮循环：
  AI 看到北京 35°C、上海 22°C → 信息够了 → tool_calls = None
  走 else → return "北京更热，35°C vs 22°C"
  循环结束
```

整个 Agent 循环就是：AI 想 → 你做 → AI 再想 → 你再做 → ... → AI 说够了 → 结束。

你已经写过这个循环了，Agent 就是在这个基础上做得更完善。

## 四、Agent 的核心组成

一个完整的 Agent 由 5 个部分组成：

```
Agent = LLM（大脑）+ Tools（工具）+ Memory（记忆）+ Planning（规划）+ Prompt（指令）
```

### 1. LLM（大脑）

Agent 的核心决策者。所有的"思考"都是 LLM 在做：
- 理解用户意图
- 决定下一步做什么
- 选择用哪个工具
- 判断任务是否完成

不同 LLM 的 Agent 能力差异很大：
- GPT-4、Claude 3.5：Agent 能力强，复杂任务表现好
- DeepSeek：性价比高，简单 Agent 任务够用
- 小模型（7B 以下）：Agent 能力弱，容易出错

### 2. Tools（工具）

Agent 能调用的所有函数/API，就是你在 Agent Skills 阶段学的 Function Calling：

```python
tools = [
    搜索工具,      # 搜索互联网
    数据库工具,    # 查询/写入数据库
    代码执行工具,  # 执行 Python 代码
    文件操作工具,  # 读写文件
    API 调用工具,  # 调用第三方 API
    RAG 检索工具,  # 从知识库检索（你学的 RAG）
]
```

工具越多，Agent 能力越强，但也越容易选错工具。

### 3. Memory（记忆）

你在 Agent Skills 阶段学的三种记忆：
- 短期记忆：当前对话的 messages 列表
- 工作记忆：滑动窗口，控制 Token 成本
- 长期记忆：持久化存储用户信息

Agent 场景下还多一个：
- 任务记忆：记住当前任务执行到哪一步了，中间结果是什么

### 4. Planning（规划）

Agent 怎么拆解复杂任务：
- ReAct：边想边做（最常用，下一篇详细讲）
- Plan-and-Execute：先想好所有步骤，再一步步执行
- 自适应规划：执行过程中根据结果调整计划

### 5. Prompt（指令）

System Prompt 定义了 Agent 的身份、能力边界和行为规范：

```python
system_prompt = """你是一个数据分析助手。

你可以使用以下工具：
- read_file: 读取数据文件
- run_python: 执行 Python 代码
- create_chart: 生成图表

工作流程：
1. 先理解用户的分析需求
2. 读取数据文件
3. 编写分析代码
4. 生成图表和报告

注意事项：
- 每一步都要解释你在做什么
- 如果数据有问题，先告诉用户
- 不要执行危险的代码"""
```

好的 System Prompt 能让 Agent 表现更稳定、更可控。

## 六、Agent 的分类

### 按自主程度分

| 类型 | 说明 | 例子 |
|------|------|------|
| 固定流程 Agent | 步骤写死，AI 只负责每步的具体执行 | 固定的数据处理 pipeline |
| 半自主 Agent | 大框架固定，细节 AI 自己决定 | ReAct Agent（最常用） |
| 全自主 Agent | AI 完全自己决定做什么 | AutoGPT（不稳定，生产环境少用） |

实际项目中最常用的是半自主 Agent（ReAct 模式），既有灵活性又可控。

### 按任务类型分

| 类型 | 说明 | 典型场景 |
|------|------|---------|
| 对话型 Agent | 主要做多轮对话 | 智能客服、个人助手 |
| 任务型 Agent | 完成具体任务 | 数据分析、代码生成 |
| 检索型 Agent | 搜索和整合信息 | RAG 问答、研究助手 |
| 创作型 Agent | 生成内容 | 写文章、生成报告 |

一个 Agent 可以同时具备多种类型的能力。

## 七、Agent 的局限性（重要）

Agent 不是万能的，了解局限才能用好它：

### 1. LLM 会犯错
AI 的推理不是 100% 正确的，可能选错工具、传错参数、做出错误判断。

应对：设置最大循环次数、关键操作加人工确认。

### 2. 成本问题
Agent 每一轮循环都是一次 API 调用，复杂任务可能调用十几次，Token 成本高。

应对：选择性价比高的模型、优化 prompt 减少不必要的循环。

### 3. 延迟问题
多轮循环意味着多次 API 调用，总耗时 = 单次耗时 × 循环次数。

应对：并行调用工具、缓存常用结果。

### 4. 安全风险
Agent 能执行代码、操作数据库、调用 API，如果 AI 判断错误可能造成破坏。

应对：工具权限控制、危险操作加确认、沙箱环境执行代码。（沙箱（Sandbox）就是一个隔离的、受限的运行环境，代码在里面跑，就算出了问题也不会影响到外面的真实系统。）

### 5. 可控性
全自主 Agent 的行为难以预测，可能做出意料之外的操作。

应对：用半自主模式（ReAct）、设置明确的行为边界、日志记录所有操作。
