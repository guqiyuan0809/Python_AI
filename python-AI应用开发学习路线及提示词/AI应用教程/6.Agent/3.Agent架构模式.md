# Agent 架构模式

> 不同的任务适合不同的 Agent 架构。
> ReAct 是最常用的，但了解其他模式能帮你在合适的场景选择合适的方案。

---

## 一、三种主要架构模式

```
1. ReAct（边想边做）        ← 最常用，上一篇已详细讲
2. Plan-and-Execute（先想后做）← 适合步骤明确的任务
3. 自适应规划（边做边调整计划）← 两者结合
```

---

## 二、Plan-and-Execute（先规划再执行）

### 核心思路

和 ReAct 的区别：ReAct 是走一步看一步，Plan-and-Execute 是先把所有步骤想好，再一步步执行。

```
ReAct：
  想 → 做 → 看 → 想 → 做 → 看 → ... → 完成

Plan-and-Execute：
  想好所有步骤 → 执行步骤1 → 执行步骤2 → ... → 完成
```

### 执行流程

```
用户：帮我分析这个 CSV 文件的销售数据，生成月度报告

━━━ Planning 阶段 ━━━
AI 制定计划：
  Step 1: 读取 CSV 文件
  Step 2: 数据清洗（去除空值和异常值）
  Step 3: 按月份汇总销售额
  Step 4: 计算同比/环比增长率
  Step 5: 生成可视化图表
  Step 6: 输出分析报告

━━━ Execution 阶段 ━━━
执行 Step 1: read_file("sales.csv") → 成功
执行 Step 2: clean_data(data) → 成功
执行 Step 3: group_by_month(data) → 成功
执行 Step 4: calculate_growth(data) → 成功
执行 Step 5: create_chart(data) → 成功
执行 Step 6: generate_report(results) → 输出报告
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 步骤清晰，用户能看到完整计划 | 计划可能不准确 |
| 执行过程可预测 | 遇到意外情况不灵活 |
| 适合步骤明确的任务 | 如果某步失败，后续步骤可能都废了 |

## 三、自适应规划（Plan + ReAct 结合）

### 核心思路

先制定一个大致计划，但执行过程中可以根据实际情况调整计划。

```
制定初始计划 → 执行步骤1 → 看结果 → 
  结果符合预期？→ 继续执行步骤2
  结果不符合？  → 修改计划，调整后续步骤
```

### 执行示例

```
用户：帮我查找并修复代码中的 bug

━━━ 初始计划 ━━━
Step 1: 读取代码文件
Step 2: 分析代码逻辑
Step 3: 定位 bug
Step 4: 修复 bug
Step 5: 测试验证

━━━ 执行 ━━━
执行 Step 1: read_file("app.py") → 成功
执行 Step 2: 分析代码... 发现有 3 个可疑位置

[调整计划] 原来以为只有 1 个 bug，现在发现 3 个可疑位置
新计划：
Step 3a: 检查第 1 个可疑位置
Step 3b: 检查第 2 个可疑位置
Step 3c: 检查第 3 个可疑位置
Step 4: 修复确认的 bug
Step 5: 测试验证

继续执行...
```

### 这是最实用的模式

实际项目中，纯 ReAct 和纯 Plan-and-Execute 都有局限：
- 纯 ReAct：没有全局视角，可能走弯路
- 纯 Plan-and-Execute：太死板，遇到意外就卡住

自适应规划结合了两者的优点：有计划但不死板，能根据实际情况灵活调整。

## 四、架构选择指南

| 场景 | 推荐架构 | 原因 |
|------|---------|------|
| 简单问答 + 工具调用 | ReAct | 步骤少，不需要复杂规划 |
| 数据分析、报告生成 | Plan-and-Execute | 步骤明确，流程固定 |
| 复杂调试、研究任务 | 自适应规划 | 需要灵活应对意外情况 |
| 对话型助手 | ReAct | 交互式，一步步来 |
| 批量处理任务 | Plan-and-Execute | 流程固定，批量执行 |

### 实际建议

刚开始做 Agent 项目，用 ReAct 就够了。等你遇到 ReAct 搞不定的场景（比如任务太复杂、AI 总是绕圈），再考虑 Plan-and-Execute 或自适应规划。

## 五、单 Agent vs 多 Agent

### 单 Agent（本章重点）

一个 Agent 具备所有能力，自己完成所有任务：

```
用户 → [Agent（多个工具）] → 结果
```

适合：大部分场景。一个 Agent 配 5-10 个工具，能处理很多任务。

### 多 Agent（第 7 章了解）

多个 Agent 各自负责不同领域，协作完成任务：

```
用户 → [调度 Agent] → [搜索 Agent]
                    → [分析 Agent]  → 汇总结果
                    → [写作 Agent]
```

适合：非常复杂的任务，单个 Agent 搞不定时。

现阶段专注单 Agent 就好，多 Agent 留到第 7 章。

## 六、Agent 的控制流

控制流描述的是 Agent 内部**步骤之间的执行顺序和跳转逻辑**。选好架构（ReAct / Plan-and-Execute）之后，还需要决定步骤怎么串起来——是一条路走到底，还是循环、分支、并行？

不管用哪种架构，Agent 的控制流都可以归纳为几种基本模式：

### 1. 线性流（Sequential）

```
步骤1 → 步骤2 → 步骤3 → 完成
```

最简单，每步固定，适合流程明确的任务。

### 2. 循环流（Loop）

```
步骤1 → 步骤2 → 判断 → 不满足 → 回到步骤1
                      → 满足   → 完成
```

ReAct 就是这种模式。

### 3. 条件分支（Conditional）

```
步骤1 → 判断条件 → 条件A → 步骤2a → 完成
                 → 条件B → 步骤2b → 完成
```

根据中间结果走不同路径。

### 4. 并行流（Parallel）

```
步骤1 → 步骤2a ─┐
      → 步骤2b ─┤→ 汇总 → 完成
      → 步骤2c ─┘
```

多个独立步骤同时执行，提高效率。

这些控制流可以组合使用，LangGraph 框架就是用图（Graph）来定义这些控制流的。

---


工作流 = 任务步骤 + 控制流 + 工具调用 + 数据传递 + 状态管理。

## 总结

- ReAct（边想边做）：最常用，90% 场景够用
- Plan-and-Execute（先想后做）：步骤明确的任务
- 自适应规划（边做边调整）：最实用，结合两者优点
- 刚开始做项目用 ReAct，遇到瓶颈再升级
- 单 Agent 是本章重点，多 Agent 留到第 7 章
