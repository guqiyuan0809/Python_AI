# Agent — 基础概念

> 你之前学的 Chain（包括 RAG 的 Retrieval Chain）是"流水线"：步骤是固定的，按顺序执行。
> Agent 是"会思考的程序"：让 LLM 自己决定下一步做什么、调用哪个工具。
> 这是 LangChain 里最强大、也最贴近"AI 助手"概念的部分。

---

## 一、Chain vs Agent — 一图看懂

### Chain（你已经会了）

```
用户输入 → 步骤1 → 步骤2 → 步骤3 → 输出
              ↑ 流程是写死的，每次都一样
```

例如 RAG Chain：用户问题 → 检索 → 拼接 Prompt → LLM 回答 → 返回。
不管用户问什么，永远走这 4 步，一步不差。

### Agent

```
用户输入 → LLM 思考"我该做什么？"
              ↓
        ┌─────────────────┐
        │  需要查天气？   │ → 调用 weather_tool
        │  需要算数学？   │ → 调用 calculator_tool
        │  需要搜资料？   │ → 调用 search_tool
        │  够了，可以回答 │ → 直接输出
        └─────────────────┘
              ↑ 流程是 LLM 临时决定的，每次可能不同
```

**核心区别**：Chain 是"工程师写好流程"，Agent 是"LLM 自己决定流程"。

---

## 二、什么时候用 Agent

| 场景 | 用 Chain 还是 Agent |
|------|------------------|
| 固定流程的 RAG 问答 | Chain ✅ |
| 用户问的问题需要"分情况处理" | Agent ✅ |
| 需要调用多个外部工具（查天气 + 算数 + 搜索） | Agent ✅ |
| 需要多步推理（先查 A，根据结果再查 B） | Agent ✅ |
| 流程清晰、不需要思考 | Chain ✅ |

**简单原则**：能写死流程就用 Chain，需要"看情况"就用 Agent。

---

## 三、Agent 的核心三件套

任何 Agent 都由三部分组成：

```
┌────────────────────────────────────┐
│  1. LLM        负责"思考"          │
│  2. Tools      负责"动手"          │
│  3. Prompt     告诉 LLM 怎么思考   │
└────────────────────────────────────┘
        ↓ 串起来
   AgentExecutor 负责"循环执行"
```

### 1. LLM — 大脑

负责理解用户问题、选择工具、决定何时停止。
要求：必须支持 **Function Calling / Tool Calling**（DeepSeek、GPT-4、Claude 都支持）。

### 2. Tools — 手脚

LLM 自己只会"说话"，不会查天气、不会算账、不会查数据库。
工具就是一个个 Python 函数，让 LLM 通过调用它们来真正"做事"。

```python
# 一个 Tool 本质就是：函数 + 描述
@tool
def get_weather(city: str) -> str:
    """查询某个城市的天气"""
    return f"{city} 今天 25 度，晴"
```

### 3. Prompt — 指挥手册

告诉 LLM："你是谁、你能做什么、你该怎么思考"。
通常包含一个特殊占位符 `agent_scratchpad`，用来记录 Agent 的"思考过程"。

## 四、Agent 的执行循环（最关键）

Agent 不是"调一次 LLM 就结束"，它是一个**循环**：

```
┌──────────────────────────────────────────┐
│           AgentExecutor 循环             │
├──────────────────────────────────────────┤
│                                          │
│  Step 1: LLM 看用户问题 + 已有信息       │
│          决定下一步：                    │
│            (a) 调用某个工具              │
│            (b) 直接回答用户              │
│                                          │
│  Step 2: 如果选 (a)，执行工具，拿到结果  │
│          把结果写入 scratchpad           │
│          回到 Step 1                     │
│                                          │
│  Step 3: 如果选 (b)，输出回答，结束      │
│                                          │
└──────────────────────────────────────────┘
```

### 一个具体例子

用户问："北京今天天气怎么样？比上海热多少度？"

```
[第 1 轮]
LLM 想：我需要先查北京天气
→ 调用 get_weather("北京") → 返回 "北京 25 度"

[第 2 轮]
LLM 想：还需要查上海
→ 调用 get_weather("上海") → 返回 "上海 22 度"

[第 3 轮]
LLM 想：信息够了，可以回答
→ 输出："北京今天 25 度，比上海高 3 度"
```

**3 次调用 LLM，2 次调用工具，整个过程是 LLM 自己规划出来的**。

## 五、两种 Agent 实现方式

### 1. Tool Calling Agent（推荐，现代方式）⭐ 必练

借助 LLM 原生的 Function Calling 能力。
LLM 直接返回结构化的"我要调用某个工具"信号，准确率高。

```python
from langchain.agents import create_tool_calling_agent, AgentExecutor
agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)
```

**适用模型**：DeepSeek、GPT-4、Claude、Qwen 等支持 Function Calling 的模型。
**优点**：稳定、准确，工具参数能严格遵守类型。

### 2. ReAct Agent（经典方式）📖 了解即可

通过 Prompt 让 LLM"演"出推理过程：
```
Thought: 我需要查天气
Action: get_weather
Action Input: 北京
Observation: 北京 25 度
Thought: 够了，可以回答
Final Answer: 北京 25 度
```

```python
from langchain.agents import create_react_agent, AgentExecutor
agent = create_react_agent(llm, tools, prompt)
```

**优点**：不要求模型支持 Function Calling，老模型也能用。
**缺点**：依赖 Prompt 解析，偶尔会"格式错误"。

**结论**：实际项目首选 `create_tool_calling_agent`，老旧模型才用 ReAct。

## 六、与你之前学的对比

### 纯代码（手动写"伪 Agent"）

```python
question = "北京天气怎么样？"

# 你得自己判断：用户想干嘛
if "天气" in question:
    city = extract_city(question)        # 自己写实体抽取
    result = get_weather(city)           # 调用接口
    answer = format_answer(result)       # 拼接回复
elif "计算" in question:
    ...
else:
    answer = call_llm(question)
```

问题：
- 只能处理你想到的情况，问题稍微一变就崩
- 多步推理（先查 A 再查 B）几乎写不了
- 加新功能要改一堆 if-else

### LangChain Agent

```python
tools = [get_weather, calculator, search]
agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)

# 用户问什么都行，LLM 自己决定怎么处理
executor.invoke({"input": "北京天气怎么样？比上海热多少度？"})
```

**LangChain Agent 帮你做了什么**：
1. 让 LLM 自己决策，省掉 if-else 地狱
2. 自动管理"思考-动手-观察"的循环
3. 加新功能 = 加一个 Tool，不用改 Agent 主体逻辑
4. 支持多步推理

## 七、完整流程（Agent 视角）

```
用户提问
   ↓
Prompt 模板（告诉 LLM 角色 + scratchpad 占位符）
   ↓
LLM 思考 → 决定调用哪个 Tool（或直接回答）
   ↓                              ↓
执行 Tool ←─── 循环 ───→     直接输出回答
   ↓
Tool 结果回填到 scratchpad
   ↓
（回到 LLM 继续思考，直到决定回答）
```

## 九、面试怎么讲 Agent

```
面试官："你说说 Chain 和 Agent 的区别？"

你：
"Chain 是固定流程的流水线，比如 RAG 系统永远是'检索→拼接→生成'三步。
Agent 是动态决策——LLM 看到用户问题后自己决定调用哪个工具、调用几次、
什么时候结束。

Agent 有三个核心组件：LLM 负责思考，Tools 是能调用的外部能力，
Prompt 告诉 LLM 角色和规则。运行时是一个循环：
LLM 决定 → 调用工具 → 把结果回填 → 再决定，直到 LLM 觉得够了为止。

实际项目我用的是 create_tool_calling_agent，因为它基于 LLM 原生的
Function Calling，比 ReAct 更稳定。"
```

## 总结

- **Agent = LLM + Tools + Prompt + 循环执行器**
- Chain 流程写死，Agent 让 LLM 自己决策
- 现代实现首选 `create_tool_calling_agent`（基于 Function Calling）
- 核心循环：思考 → 调用工具 → 观察结果 → 再思考 → ... → 输出
