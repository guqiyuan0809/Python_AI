# Tools — 工具定义

> 上一章你知道了 Agent = LLM + Tools + Prompt。
> 这一章解决：**怎么写 Tool？怎么让 LLM 看懂这个 Tool？**
> 这是 Agent 工程中你写代码最多的地方。

---

## 一、Tool 是什么

一个 Tool 本质上就是：

```
一个 Python 函数 + 一段描述
```

- **函数本身**：实际干活的代码（查天气、查数据库、做计算）
- **描述**：告诉 LLM "这个工具是干嘛的、参数怎么填"

LLM 看不到你的函数代码，**它只能看 description**。所以描述写得好不好，决定了 Agent 用得准不准。

---

## 二、最简单的方式：@tool 装饰器 ⭐ 必练

LangChain 提供了 `@tool` 装饰器，能把普通函数一键变成 Tool。

### 基本写法

```python
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """查询某个城市的实时天气。

    Args:
        city: 城市名称，例如"北京"、"上海"
    """
    # 真实项目里这里会调用气象 API
    return f"{city} 今天 25 度，晴朗"
```

**就这么简单**。`@tool` 装饰器会自动从函数中提取：

| 提取自 | 提供给 LLM |
|--------|-----------|
| 函数名 `get_weather` | Tool 名字 |
| 文档字符串（docstring） | Tool 描述 |
| 类型注解 `city: str` | 参数类型 |

### 查看自动生成的"工具说明书"

```python
print(get_weather.name)         # 'get_weather'
print(get_weather.description)  # '查询某个城市的实时天气。\n...'
print(get_weather.args)         # {'city': {'title': 'City', 'type': 'string'}}
```

LLM 看到的就是这些信息，根据它来决定要不要调用、参数填什么。

### 调用 Tool

```python
# 你也可以手动调用，用于测试
result = get_weather.invoke({"city": "北京"})
print(result)   # '北京 今天 25 度，晴朗'

# 实际项目中不需要手动调，AgentExecutor 会自动调
```

## 三、写好 docstring 的三个原则 ⭐ 重点

LLM 选不选你的 Tool，参数填不填对，**全靠 docstring**。这是 Agent 工程最关键的一点。

### 原则 1：第一句话说清楚"这个工具能干嘛"

```python
# ❌ 模糊
"""一个工具"""

# ❌ 太技术，LLM 看不懂用户场景
"""调用 X-API 接口"""

# ✅ 清晰，LLM 一看就知道什么时候用
"""查询指定城市当前的实时天气，包含温度和天气状况"""
```

### 原则 2：每个参数单独说明

```python
@tool
def search_flight(from_city: str, to_city: str, date: str) -> str:
    """查询机票信息。

    Args:
        from_city: 出发城市，例如"北京"
        to_city: 到达城市，例如"上海"
        date: 出发日期，格式为 YYYY-MM-DD，例如"2026-05-01"
    """
    ...
```

**关键**：一定要给"格式示例"，否则 LLM 会乱传（比如把日期写成"明天"、"五一"）。

### 原则 3：写清楚"什么时候不用这个工具"

```python
@tool
def calculator(expression: str) -> str:
    """执行数学计算（只支持加减乘除）。

    Args:
        expression: 数学表达式，例如 "2 + 3 * 4"

    注意：不要用于查天气、查日期等非数学任务。
    """
    ...
```

如果你的工具有局限性，写在 description 里能减少 LLM 误用。

## 四、复杂参数：用 Pydantic 定义参数模式

当参数多、有验证规则时，单纯靠类型注解不够，可以用 Pydantic：

```python
from pydantic import BaseModel, Field
from langchain_core.tools import tool

class FlightSearchInput(BaseModel):
    from_city: str = Field(..., description="出发城市，例如'北京'")
    to_city: str = Field(..., description="到达城市，例如'上海'")
    date: str = Field(..., description="日期，格式 YYYY-MM-DD")
    cabin: str = Field(default="经济舱", description="舱位，可选'经济舱'/'商务舱'/'头等舱'")

@tool(args_schema=FlightSearchInput)
def search_flight(from_city: str, to_city: str, date: str, cabin: str = "经济舱") -> str:
    """查询机票信息"""
    return f"找到 {from_city}→{to_city} {date} {cabin}的航班"
```

**好处**：
- 参数描述更结构化
- 可以加默认值、可选参数
- LLM 选择参数会更准

## 五、自定义 Tool 名字和描述

有时候函数名/docstring 不方便改，可以在装饰器上覆盖：

```python
@tool("weather_query", description="查询天气，输入城市名返回天气状况")
def get_weather_internal_v2(city: str) -> str:
    return f"{city} 今天 25 度"

print(get_weather_internal_v2.name)         # 'weather_query'
print(get_weather_internal_v2.description)  # '查询天气，输入城市名返回天气状况'
```

## 六、内置工具 — LangChain 自带的"现成 Tool"

LangChain 提供了一些常用的内置工具，可以直接拿来用：

### 1. 搜索类

```python
# DuckDuckGo 搜索（免费）
# pip install duckduckgo-search
from langchain_community.tools import DuckDuckGoSearchRun
search = DuckDuckGoSearchRun()
print(search.invoke("今天有什么新闻"))

# Tavily（更专业的 AI 搜索，需要 API key）
# pip install tavily-python
from langchain_community.tools.tavily_search import TavilySearchResults
tavily = TavilySearchResults(max_results=3)
```

### 2. 维基百科

```python
# pip install wikipedia
from langchain_community.tools import WikipediaQueryRun
from langchain_community.utilities import WikipediaAPIWrapper

wiki = WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper())
print(wiki.invoke("Python 编程语言"))
```

### 3. Python REPL（让 Agent 跑 Python 代码）

```python
from langchain_experimental.tools import PythonREPLTool

python_repl = PythonREPLTool()
print(python_repl.invoke("print(2 ** 10)"))   # 1024
```

⚠️ **危险**：这相当于把代码执行权给 LLM，不要在生产环境随便用。

### 4. 把现成的工具当作工具列表

```python
from langchain_community.tools import DuckDuckGoSearchRun
from langchain_experimental.tools import PythonREPLTool

tools = [
    DuckDuckGoSearchRun(),
    PythonREPLTool(),
    get_weather,        # 你自己写的 @tool
    search_flight,
]
```

**注意**：自定义工具和内置工具可以混着用。

## 七、把 Retriever 也变成 Tool — 让 Agent 用 RAG ⭐ 实战

你之前学的 RAG（Retrieval Chain）也能包装成 Tool 给 Agent 用：

```python
from langchain.tools.retriever import create_retriever_tool

# 假设你已经有了 retriever（来自上一章 Vector Stores）
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# 包装成 Tool
rag_tool = create_retriever_tool(
    retriever,
    name="company_knowledge_base",
    description="搜索公司内部文档，包括产品手册、流程规范等。当用户询问公司相关信息时使用此工具。"
)

tools = [rag_tool, get_weather, calculator]
```

**这是 Agent + RAG 的标准用法**：
- Agent 看到用户问题，自己决定要不要查知识库
- 不该查的时候直接回答（比如"今天天气怎么样"），不浪费检索
- 该查的时候自动调用 RAG，得到结果后再生成回答

## 八、错误处理 — 让 Tool 优雅失败

工具调用可能失败（网络超时、参数非法等）。要让 Agent 能"接住"错误继续推理：

```python
@tool
def get_stock_price(symbol: str) -> str:
    """查询股票当前价格"""
    try:
        # 模拟可能失败的 API 调用
        if not symbol.isalpha():
            return f"错误：股票代码 '{symbol}' 不合法，应该是字母"
        return f"{symbol} 当前 $100"
    except Exception as e:
        return f"查询失败：{e}，请重试"
```

**关键**：**错误信息也作为字符串返回，不要 raise 异常**。
LLM 看到"错误：股票代码不合法"，就会自己修正参数重试一次。

---

## 九、ToolException — 让框架来处理错误

如果你想让 AgentExecutor 自己处理错误，可以抛出 `ToolException`：

```python
from langchain_core.tools import ToolException

@tool
def divide(a: float, b: float) -> float:
    """两个数相除"""
    if b == 0:
        raise ToolException("除数不能为 0")
    return a / b
```

然后在 AgentExecutor 里设置：
```python
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    handle_parsing_errors=True,
)
```

不过最简单粗暴的做法还是返回字符串，更可控。

## 十、与你之前学的对比

### 纯代码（手动调用 Function Calling）

```python
# 自己写 Tool 定义
tools_schema = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名"}
            },
            "required": ["city"]
        }
    }
}]

# 自己处理 LLM 返回的 tool_calls
response = client.chat.completions.create(
    messages=messages,
    tools=tools_schema,
)
if response.choices[0].message.tool_calls:
    tc = response.choices[0].message.tool_calls[0]
    if tc.function.name == "get_weather":
        args = json.loads(tc.function.arguments)
        result = get_weather(args["city"])
        # 还要自己拼接消息发回 LLM...
```

### LangChain @tool

```python
@tool
def get_weather(city: str) -> str:
    """查询天气"""
    return f"{city} 25 度"

# 完了。AgentExecutor 自动帮你处理调用、错误、循环
```

**LangChain 帮你做了什么**：
1. 自动从 docstring + 类型注解生成 JSON Schema
2. 自动处理工具调用循环（LLM 选 → 调工具 → 把结果回填 → 再选）
3. 错误处理、重试、超时统一管控
4. 工具列表传给 Agent 即可，不用手动管 schema

## 十一、Tool 设计的最佳实践

```
1. 一个 Tool 干一件事，不要把所有功能塞一个 Tool
   ❌ "万能查询工具：查天气/股票/新闻"
   ✅ get_weather / get_stock_price / search_news 分开

2. Tool 的输入输出都用字符串或简单类型
   LLM 难以填复杂嵌套对象

3. docstring 要给清楚使用场景和示例
   LLM 不会读你的代码，只会读 description



   docstring 是你写在函数里的文档字符串
   description 是 LangChain 生成出来、给 LLM 看的工具说明

4. 失败时返回错误字符串，不要 raise
   让 LLM 能根据错误信息自己修正

5. 工具数量控制在 5~10 个内
   太多会让 LLM 选择困难，准确率下降
```

## 十二、面试怎么讲 Tools

```
面试官："你给 Agent 写工具时注意了什么？"

你：
"几个要点：
1. docstring 写得很详细，因为 LLM 只看描述决定要不要调，
   尤其参数描述要给格式示例，比如日期写成 YYYY-MM-DD。

2. 一个工具只做一件事，不会把多个功能合并。比如查天气和查股票
   一定是两个独立的 @tool。

3. 错误用字符串返回而不是抛异常，让 LLM 能根据错误信息自己修正
   再调一次。

4. RAG 也是一个 Tool —— 用 create_retriever_tool 把检索器包起来
   交给 Agent，比直接用 Retrieval Chain 更灵活，因为 Agent 知道
   什么时候该查、什么时候不该查。"
```

---

## 总结

- Tool = 函数 + docstring，LLM 只看 docstring 决定调不调
- `@tool` 装饰器是最简单的方式，自动从类型注解 + docstring 生成 schema
- 复杂参数用 Pydantic + `args_schema`
- LangChain 自带很多内置 Tool（搜索、Wiki、Python REPL）
- RAG 也能包装成 Tool（`create_retriever_tool`）—— 这是 Agent + RAG 的标准做法
- 错误优先返回字符串，让 LLM 能自我修正
