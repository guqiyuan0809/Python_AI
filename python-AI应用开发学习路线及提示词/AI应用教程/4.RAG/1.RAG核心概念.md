# RAG 核心概念

## RAG 是什么？

RAG（Retrieval-Augmented Generation，检索增强生成）是目前市场需求最大的 AI 应用方向。
Retrieval /rɪˈtriːvəl/

### 核心理念

**RAG 的本质：给 AI 补充它不知道的数据，让它基于你的数据回答问题。**

用更直白的话说：
- AI 本身只知道训练时学到的通用知识
- 但它不知道你公司的产品文档、你的个人笔记、你的代码库
- RAG 就是在用户提问时，先从你的数据里找相关内容，然后"喂"给 AI
- AI 看着这些内容回答，就像开卷考试一样

---

## 为什么需要 RAG？

### 传统 AI 的问题

- 只能回答训练数据中的内容
- 不知道你公司的内部文档、最新产品信息
- 容易"胡说八道"（幻觉问题）
- 知识更新需要重新训练模型，成本高

### RAG 的解决方案

```
用户提问 
  ↓
文本向量化（Embedding）
  ↓
在向量数据库中搜索相似文档
  ↓
把找到的文档 + 用户问题一起发给 AI
  ↓
AI 基于文档内容生成回答
```

### RAG 的核心价值

1. **针对性强**：回答基于你的真实数据，不是 AI 瞎编的
2. **可控性高**：你能看到 AI 引用了哪些文档，回答有据可查
3. **实时更新**：数据更新了，AI 的回答也跟着更新，不需要重新训练模型
4. **降低幻觉**：AI 有了参考资料，不容易胡说八道
5. **数据隐私**：私有数据不需要上传训练，只在检索时使用

## RAG 的实际应用场景

### 场景 1：企业内部知识库
- 员工问："报销流程是什么？"
- RAG 从公司制度文档中找到报销相关段落
- AI 基于文档回答，而不是猜

### 场景 2：技术文档助手
- 开发者问："这个 API 怎么调用？"
- RAG 从项目的 API 文档中找到相关接口说明
- AI 给出准确的调用示例

### 场景 3：个人知识管理
- 你问："我之前学的 FastAPI 路由怎么写？"
- RAG 从你的学习笔记中找到相关内容
- AI 帮你回忆起来

### 场景 4：智能客服机器人
- 用户问："你们的退货政策是什么？"
- RAG 从产品手册中找到退货条款
- AI 给出标准答案，不会乱承诺

### 场景 5：代码库问答
- 问："这个项目的认证逻辑在哪里？"
- RAG 从代码库中找到相关文件
- AI 解释认证流程和代码位置

## RAG 的三大核心概念

### 1. Embedding（文本向量化）

把文字变成数字列表，让计算机能计算"相似度"：

```python
"你好" → [0.2, 0.8, 0.1, ...]  (实际有几百到几千维)
"你好吗" → [0.3, 0.7, 0.2, ...]
"天气真好" → [0.1, 0.2, 0.9, ...]
```

**核心原理**：数字越接近，意思越相似。

**实现方式**：调用 Embedding API，还是调用API完成，不需要写代码

```python
from openai import OpenAI
client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="你好"
)
vector = response.data[0].embedding  # 得到向量
```

**常用 Embedding 模型**：
- OpenAI：`text-embedding-3-small`（便宜）、`text-embedding-3-large`（效果好）
- 开源模型：`bge-large-zh`（中文效果好）、`m3e-base`
- 本地模型：可以用 Ollama 运行开源 Embedding 模型

### 2. 向量数据库（Vector Database）

专门存储和搜索向量的数据库。

**核心功能**：
```python
# 存入文档
db.add(text="公司年假政策：...", vector=[0.2, 0.8, ...])

# 搜索相似文档
results = db.search(query_vector=[0.3, 0.7, ...], top_k=3)
```

**常用向量数据库**：

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| Chroma | 轻量级，易上手 | 学习、小项目 |
| Milvus | 生产级，性能强 | 企业级应用 |
| Pinecone | 云服务，开箱即用 | 快速上线（收费）|
| FAISS | Facebook 开源，性能好 | 需要自己管理 |
| Qdrant | Rust 编写，性能好 | 生产环境 |

**选型建议**：
- 学习阶段：Chroma（最简单）
- 生产环境：Milvus 或 Qdrant
- 快速验证：Pinecone（但要花钱）

### 3. 文档处理 Pipeline

```
原始文档（PDF/Word/Markdown）
  ↓
文档解析（提取文本）
  ↓
文本分块（Chunking）← 重要！
  ↓
向量化（Embedding）
  ↓
存入向量数据库
```

**为什么要分块？**
- AI 的上下文窗口有限（比如 4K tokens）
- 一整本书塞不进去
- 分成小段（比如 500 字一段），只检索相关的段落
- 提高检索精度（小块更容易匹配用户问题）

**分块策略**：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 固定长度分块 | 每 500 字切一段 | 简单 | 可能切断语义 |
| 按段落分块 | 保持段落完整 | 语义完整 | 长度不均 |
| 递归分块 | 先按章节，再按段落 | 层次清晰 | 实现复杂 |
| 重叠分块 | 相邻块有 50-100 字重叠 | 避免信息断裂 | 存储冗余 |

**推荐策略**：重叠分块（chunk_size=500, overlap=50）

## RAG 的完整工作流程

### 离线阶段（准备知识库）

```python
# 1. 读取文档
docs = load_documents("./docs")

# 2. 分块
chunks = split_documents(docs, chunk_size=500, overlap=50)

# 3. 向量化
embeddings = embed_documents(chunks)

# 4. 存入向量数据库
vector_db.add(chunks, embeddings)
```

### 在线阶段（用户提问）

```python
# 1. 用户提问
question = "公司年假政策是什么？"

# 2. 问题向量化
query_vector = embed_text(question)

# 3. 检索相关文档（相似度搜索）
relevant_docs = vector_db.search(query_vector, top_k=3)

# 4. 构造 Prompt
prompt = f"""
根据以下文档回答问题：

文档内容：
{relevant_docs}

问题：{question}

请基于文档内容回答，如果文档中没有相关信息，请说"文档中没有找到相关信息"。
"""

# 5. 调用 AI 生成回答
response = llm.chat(prompt)
```

## RAG 的两大主流框架

### 1. LangChain

**特点**：
- 功能全面，生态丰富
- 支持多种数据源、向量数据库、LLM
- 社区活跃，文档完善
- 但比较复杂，学习曲线陡

**适用场景**：
- 需要复杂的 RAG pipeline
- 需要集成多种工具
- 生产环境

### 2. LlamaIndex

**特点**：
- 专注 RAG，API 更简洁
- 上手快，10 行代码跑通 demo
- 文档清晰，示例丰富
- 功能相对 LangChain 少一些

**适用场景**：
- 快速原型开发
- 简单的 RAG 应用
- 学习 RAG 概念

### 学习建议

1. **先用 LlamaIndex 快速上手**：理解 RAG 的核心流程
2. **再学 LangChain 深入理解**：掌握更多高级特性
3. **两个框架都要会**：面试时都会问到

## RAG 的关键技术点

### 1. 检索策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 向量检索 | 基于语义相似度 | 通用场景 |
| 关键词检索 | 基于 BM25 算法 | 精确匹配 |
| 混合检索 | 向量 + 关键词 | 提高召回率 |
| 重排序（Rerank）| 对检索结果二次排序 | 提高精度 |

### 2. 多轮对话记忆

用户可能会追问：
```
用户："公司年假政策是什么？"
AI："员工每年有 10 天年假..."
用户："那病假呢？"  ← 需要记住上下文
```

**解决方案**：
- 保存对话历史
- 把历史对话也加入检索上下文

### 3. 引用来源

让 AI 告诉你答案来自哪个文档：
```
AI："根据《员工手册》第 3 章，年假政策为..."
```

**实现方式**：
- 在 Prompt 中要求 AI 标注来源
- 返回检索到的文档元数据（文件名、页码）

### 4. 流式输出

让回答像打字一样逐字显示，提升用户体验：
```python
for chunk in llm.stream(prompt):
    print(chunk, end="")
```

## RAG vs 微调（Fine-tuning）

### 微调是什么？和训练大模型的区别

微调不是训练大模型，是**在已有大模型基础上做小范围调整**：

```
训练（Pre-training）：从零开始，用海量数据训练出一个大模型
                      → 需要几千张 GPU，几百万美元，几个月时间
                      → OpenAI、Google、DeepSeek 这些公司干的事

微调（Fine-tuning）：  拿一个已经训练好的大模型，用你自己的小数据再训练一下
                      → 让它在特定领域表现更好（比如说医疗术语、法律文书的风格）
                      → 需要几张 GPU，几百到几千元，几小时到几天
```

简单类比：

> 训练 = 把一个婴儿培养成大学生（从零开始）
> 微调 = 让大学生去考一个专业证书（在已有能力上补充特定知识）

### RAG vs 微调对比

| 对比项 | RAG | 微调 |
|--------|-----|------|
| 数据更新 | 实时更新，改文档即可 | 需要重新训练 |
| 成本 | 低（只需存储和检索）| 高（需要 GPU 训练）|
| 可解释性 | 高（能看到引用来源）| 低（黑盒）|
| 适用场景 | 知识库问答、文档检索 | 特定领域语言风格 |
| 技术门槛 | 低 | 高 |

**结论**：大部分 AI 应用开发岗位只需要会 RAG，不需要会微调。微调是算法岗的事。
