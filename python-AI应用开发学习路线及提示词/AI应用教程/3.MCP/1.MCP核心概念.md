# MCP — Model Context Protocol

> MCP 是 Anthropic（Claude 的公司）在 2024 年底推出的开放协议，让 AI 模型能连接外部工具和数据源。
> 你可以把它理解为 **AI 世界的 USB 接口** — 统一的标准，插上就能用。

---

## 一、MCP 解决什么问题

在 Skills 阶段，AI 只能根据你发的文字回答问题，它无法：
- 读取你电脑上的文件
- 查询数据库
- 调用第三方 API（天气、搜索、GitHub）
- 执行代码

这些能力本质上都是你已经会的技术，MCP 只是把它们标准化，让 AI 能自动调用：

| MCP 能力 | 本质 | 对应的 Python 技术 |
|----------|------|-------------------|
| 读文件 | 文件 I/O | `open()`, `os`, `pathlib` |
| 查数据库 | 数据库操作 | `sqlite3`, Django ORM, SQLAlchemy |
| 调第三方 API | HTTP 请求 | `requests`, `httpx` |
| 执行代码 | 运行脚本 | `subprocess`, `exec()` |

**你其实已经在用 MCP 了**：Windsurf / Cursor 这类 AI 编程工具就是 MCP Host，内置的 AI（Cascade）通过 MCP 能读写你的文件、执行终端命令、搜索代码、编辑代码。你每天用 Windsurf 写代码的过程，就是 MCP 的实际应用。

**MCP 就是让 AI "长出手脚"，能操作外部世界。**

```
没有 MCP：
用户 → AI → 只能用训练数据回答（不能访问外部信息）

有了 MCP：
用户 → AI → MCP → 读文件、查数据库、调 API、执行代码...
```

## 二、MCP 架构（三个角色）

### MCP 和 HTTP 的类比

**MCP 和 HTTP 一样，都是协议，而且架构非常相似**：

| HTTP 架构 | MCP 架构 | 说明 |
|-----------|----------|------|
| **浏览器** | **MCP Host（Windsurf/Cursor）** | 你直接交互的应用 |
| **浏览器内置的 HTTP 引擎** | **MCP Client** | 内置在应用里，处理协议通信 |
| **Web 服务器** | **MCP Server** | 提供具体功能 |
| **HTTP 协议** | **MCP 协议** | 通信标准 |

```
HTTP：
你 → 浏览器（内置HTTP引擎）→ HTTP协议 → Web服务器 → 返回网页

MCP：
你 → Windsurf（内置MCP Client）→ MCP协议 → MCP Server → 返回工具执行结果
```

### 三个角色详解

```
┌─────────────┐     ┌─────────────┐     ┌─────────────────┐
│  MCP Host    │ ←→  │  MCP Client  │ ←→  │  MCP Server      │
│ （AI 应用）   │     │ （协议客户端） │     │ （工具提供者）    │
└─────────────┘     └─────────────┘     └─────────────────┘
                                              │
                                              ↓
                                        外部工具/数据
                                     （文件系统、数据库、API）
```

### 三个角色详解（以 Windsurf 为例）

| 角色 | 是什么 | 在 Windsurf 中对应 | USB 类比 | HTTP 类比 |
|------|--------|-------------------|---------|----------|
| **MCP Host** | 运行 AI 的应用程序 | Windsurf 编辑器本身 | 电脑本身 | 浏览器 |
| **MCP Client** | Host 内部的协议客户端，负责和 Server 通信，相当于基于 MCP 协议的"MCP 引擎" | Windsurf 内置的通信模块（你看不到，自动工作） | USB 控制器 | HTTP 引擎 |
| **MCP Server** | 提供具体工具能力的服务 | 文件读写、终端执行、代码搜索等工具 | U 盘、打印机 | Web 服务器 |

**你在 Windsurf 中的一次操作流程**：
```
你说："帮我读取 api_demo.py"

Windsurf（Host）接收你的请求
    ↓
Cascade AI 判断需要读文件（Cascade 是 Windsurf 内置的 AI 助手，类似 Cursor 里的 Claude/GPT）
    ↓
MCP Client 发起请求给 MCP Server
    ↓
MCP Server（文件系统工具）执行读文件操作
    ↓
文件内容返回给 AI
    ↓
AI 基于文件内容回答你
```

**你平时看到的 AI 读文件、跑命令、改代码，背后就是这三个角色在协作。**

### 核心理解

**MCP 就是 AI 世界的 HTTP**：
- HTTP 让浏览器能访问全世界的网站
- MCP 让 AI 应用能使用全世界的工具

**关键相似点**：
1. **都是协议**：定义了客户端和服务端如何通信
2. **都有客户端和服务端**：HTTP 有浏览器和 Web 服务器，MCP 有 Host/Client 和 Server
3. **都是标准化的**：HTTP 让任何浏览器都能访问任何网站，MCP 让任何 AI 应用都能使用任何工具
4. **客户端内置在应用里**：浏览器内置 HTTP 引擎，Windsurf 内置 MCP Client

**简单理解**：你写一个 MCP Server，它就像一个"网站"，任何支持 MCP 的 AI 应用（就像浏览器）都能"访问"它。

**注意**：MCP Host 和 MCP Client 一般是在一起的——Host 是整个应用（如 Windsurf），Client 是 Host 内置的 MCP 引擎。你不需要单独安装或启动 Client，它随 Host 自动运行。

## 三、类比总结

### USB 类比（硬件视角）
- MCP = USB 标准
- MCP Host = 电脑
- MCP Server = U 盘、打印机等外设
- 插上就能用，不需要每个电脑单独适配

### HTTP 类比（协议视角）
- MCP = HTTP 协议
- MCP Host = 浏览器
- MCP Server = Web 服务器
- 任何浏览器都能访问任何网站

**两个类比都对**：
- USB 类比强调"即插即用"的便利性
- HTTP 类比强调"客户端-服务端"的协议本质

## 四、类比 Web 架构理解 MCP

| Web 架构 | MCP 架构 | 说明 |
|----------|----------|------|
| **浏览器** | **MCP Host（Windsurf）** | 你直接交互的应用程序 |
| **浏览器内置的 HTTP 引擎** | **MCP Client** | 内置在应用里，处理协议通信，你看不到 |
| **Web 服务器** | **MCP Server** | 提供具体功能 |
| **HTTP 协议** | **MCP 协议** | 通信标准 |

```
Web：
你 → 浏览器（内置HTTP引擎）→ HTTP协议 → Web服务器 → 返回网页

MCP：
你 → Windsurf（内置MCP Client）→ MCP协议 → MCP Server → 返回工具执行结果
```

**注意**：Host 这个词容易和 Web 里的"主机地址"搞混，但在 MCP 里 Host 的意思是"宿主应用"——AI 住在哪个应用里，那个应用就是 Host。

## 五、MCP Server 提供什么

一个 MCP Server 可以向 AI 暴露三种能力：

| 能力 | 说明 | 示例 |
|------|------|------|
| **Tools（工具）** | AI 可以调用的函数 | 查天气、发邮件、查数据库、执行代码 |
| **Resources（资源）** | AI 可以读取的数据 | 文件内容、数据库记录、API 返回的数据 |
| **Prompts（提示词模板）** | 预定义的提示词 | 代码审查模板、翻译模板 |

**最重要的是 Tools** — 这是 MCP 的核心，让 AI 能调用你定义的函数。本质就是自己编写需要的 MCP Server（工具函数），通过 MCP 协议暴露出去，AI 就能自动使用。

## 六、MCP 和 Function Calling 的关系

你在 Skills 阶段学过，OpenAI 有 Function Calling（工具调用）。MCP 和它的关系是：

| | Function Calling | MCP |
|--|--|--|
| 定义者 | OpenAI | Anthropic（开放标准） |
| 范围 | 只在一个 API 调用里用 | 跨应用、跨模型的通用协议 |
| 工具定义 | 写在 API 请求里 | 独立的 MCP Server 进程 |
| 复用性 | 每个应用要自己实现 | 写一次，所有 MCP Host 都能用 |

**类比**：
- Function Calling = 给一台电脑装了一个专属外设
- MCP = USB 标准，任何电脑插上就能用

**MCP 不替代 Function Calling，而是在它之上提供了一层标准化封装。**

**Function Calling 是什么**：OpenAI API 里的一个功能，你在代码中调用 API 时，通过 `tools` 参数告诉 AI 有哪些函数可用，AI 会判断什么时候调用。它能让 AI 调用**任何你定义的 Python 函数**（查数据库、读文件、调 API、做计算等），调第三方 API 只是其中功能之一。Function Calling 是开发者写自己的 AI 应用时使用的功能，后面学 Agent（第 6 阶段）会大量实践。

**注意：`tools` 参数和 MCP Server 是两套独立的体系，不是嵌套关系**：

```
体系一：OpenAI Function Calling（代码级别）
你的代码 → tools 参数定义函数 → AI 判断调用哪个 → 你的代码执行函数

体系二：MCP（应用级别）
MCP Host（Windsurf）→ MCP Client → MCP Server 提供工具 → AI 调用
```

两者目标一样（让 AI 能调用外部工具），实现方式不同，但从功能强度来说 **MCP Server 更强**——它是独立进程、可跨应用复用，而 Function Calling 的工具只在当次 API 调用中有效：

| | Function Calling | MCP |
|--|--|--|
| 工具定义在哪 | `tools` 参数里（函数描述） | MCP Server 进程里 |
| 谁来对接 | 你自己写代码 | MCP 协议自动对接 |
| 适用场景 | 自己写 AI 后端 | 给现成 AI 应用加功能 |

## 七、MCP 的通信方式

MCP Client 和 Server 之间有两种通信方式：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **stdio（标准输入输出）** | Server 作为子进程运行，通过 stdin/stdout 通信 | 本地工具，如文件系统、本地数据库 |
| **SSE（Server-Sent Events）** | Server 作为独立 HTTP 服务运行 | 远程服务、需要多客户端共享的场景 |

**大部分场景用 stdio 就行**，Server 跑在本地，简单直接。

## 九、MCP 和你的学习路线的关系

```
Skills（API 调用）— AI 能回答问题
    ↓
MCP（连接外部工具）— AI 能操作外部世界  ← 你在这里
    ↓
RAG（知识库）— AI 能基于你的文档回答
    ↓
Agent（智能体）— AI 能自主决策 + 调用工具 + 完成复杂任务
```

**MCP 是 Agent 的前置知识**：Agent 需要调用工具，MCP 提供了标准化的工具调用方式。理解 MCP 后，后面学 Agent 会更顺畅。

## 十、为什么 MCP 阶段偏理论

- **MCP 本身是一个协议标准**，核心是理解架构和概念（Host → Client → Server），而不是写大量代码
- **实际使用中**，大部分时候是用现成的 MCP Server（文件系统、GitHub、数据库等），不需要从零写
- **自己写 MCP Server 代码量很少**（`@mcp.tool()` 装饰器 + 函数），重点在理解它的设计思路

---

## 十一、总结

- **MCP 是什么**：AI 连接外部工具的标准协议（AI 的 USB 接口）
- **核心价值**：写一个 MCP Server，所有支持 MCP 的 AI 应用都能用
- **三个角色**：Host（AI 应用）→ Client（协议客户端）→ Server（工具提供者）
- **最重要的能力**：Tools（让 AI 调用你的函数）
- **通信方式**：stdio（本地）或 SSE（远程）
- **和 Function Calling 的关系**：MCP 是在 Function Calling 之上的标准化封装
