日常开发中可以使用训练即可


# 提示词工程（Prompt Engineering）


> 提示词的本质：给 AI 提供上下文和指令，形式不限，关键是表达清晰、信息充分。

---

## 1. 直接对话（最常见）

实时输入问题，和 AI 交互式沟通。

```
示例：
"这段代码的 hasattr 是什么意思？"
"帮我把这个项目 Docker 化部署到阿里云"
```

**适用场景**：日常开发、调试、学习

---

## 2. 预写文档（.md / .txt）

提前写好需求、背景、规则，让 AI 读取后再工作。

```
示例：
- 模块业务逻辑.md → 描述项目所有模块的功能和接口
- 部署文档.md → 描述部署流程和配置
```

**适用场景**：复杂项目、需要 AI 理解大量背景信息

---

## 3. System Prompt（系统提示词）,关键就是给API设定一个角色

在 API 调用时设置的"角色设定"，用户看不到但一直生效。

```
你是一个资深 Python 后端工程师，擅长 Django 和 FastAPI。
回答要简洁，用中文，给出代码示例。
不要解释基础概念，直接给解决方案。
```

**适用场景**：开发 AI 应用时，定义 AI 的行为和风格

---

## 4. Few-shot 示例（少样本学习）用于格式化输出

给 AI 几个"输入→输出"的例子，让它学会你要的格式。

```
示例：
输入：用户注册  →  输出：POST /api/auth/register
输入：用户登录  →  输出：POST /api/auth/login
输入：创建预约  →  输出：？

AI 会自动推断：POST /api/appointments/
```

**适用场景**：需要 AI 按固定格式输出、批量生成内容

---

## 5. 模板/框架

固定结构的提示词模板，每次填空就行。

```
角色：你是___
背景：___
任务：___
输出格式：___
约束：___
```

常见框架：
- **CRISPE**：Capacity（角色）、Request（请求）、Insight（背景）、Statement（指令）、Personality（风格）、Experiment（实验）
- **CO-STAR**：Context（背景）、Objective（目标）、Style（风格）、Tone（语气）、Audience（受众）、Response（输出格式）

**适用场景**：写复杂提示词时，用框架确保信息完整

---

## 6. Workflow 文件（工作流），工作流本身也是一种提示词

定义自动化工作流程，AI 按步骤执行。

```markdown
# 部署流程
1. 读取项目配置
2. 创建 Dockerfile
3. 创建 docker-compose.yml
4. 上传到服务器
5. 启动容器
```

**适用场景**：重复性任务自动化、团队协作规范

---

## 7. 代码注释 / Docstring

写代码时的注释本身也是提示词，AI 补全代码时会参考。

```python
# 根据用户角色返回不同的查询集：咨询师看所有预约，普通用户只看自己的
def get_queryset(self):
    # AI 会根据这个注释生成对应逻辑
```

**适用场景**：AI 辅助编程（Copilot、Cascade）

---

## 总结

| 类型 | 形式 | 适用场景 |
|------|------|---------|
| **直接对话** | 实时输入 | 日常开发、调试 |
| **预写文档** | .md / .txt 文件 | 复杂项目背景 |
| **System Prompt** | API 中的角色设定 | 开发 AI 应用 |
| **Few-shot 示例** | 输入→输出样例 | 固定格式输出 |
| **模板/框架** | 结构化填空 | 复杂提示词 |
| **Workflow** | 步骤化流程 | 自动化任务 |
| **代码注释** | 注释/Docstring | AI 辅助编程 |

**核心原则**：不管用哪种形式，好的提示词 = **清晰的意图 + 充分的上下文 + 明确的约束**。
