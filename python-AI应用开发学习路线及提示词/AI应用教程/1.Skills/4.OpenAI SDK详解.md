# OpenAI SDK 详解

> 你代码里的 `from openai import OpenAI`，这个 `openai` 到底是什么？

---

## openai 是什么

`openai` 是 OpenAI 公司发布的 Python SDK（软件开发工具包），用来调用大模型 API。

```bash
pip install openai
```

它本质上就是一个封装好的 HTTP 客户端，帮你：
- 构造请求（拼接 URL、设置 Headers、序列化 JSON）
- 发送请求（POST 到大模型 API）
- 解析响应（把 JSON 转成 Python 对象）

**你完全可以用 requests 库手动调，但 openai SDK 更方便。**

---

## 为什么所有模型都用 openai 这个库

因为 OpenAI 定义了一套 API 格式标准（Chat Completions API），现在几乎所有大模型厂商都兼容这个格式：

```
OpenAI  → 原生支持
DeepSeek → 兼容 OpenAI 格式
通义千问 → 兼容 OpenAI 格式
Ollama  → 兼容 OpenAI 格式
OpenRouter → 兼容 OpenAI 格式
```

所以你只需要学一个 SDK，换模型只改两个参数：

```python
from openai import OpenAI

# 调用 OpenAI
client = OpenAI(api_key="sk-xxx")

# 调用 DeepSeek —— 只改 base_url 和 api_key
client = OpenAI(api_key="sk-xxx", base_url="https://api.deepseek.com")

# 调用 OpenRouter —— 只改 base_url 和 api_key
client = OpenAI(api_key="sk-or-xxx", base_url="https://openrouter.ai/api/v1")

# 调用本地 Ollama —— 只改 base_url
client = OpenAI(api_key="ollama", base_url="http://localhost:11434/v1")
```

**一套代码，调所有模型。这就是为什么大家都用 openai 这个库。**

---

## 核心概念：Chat Completions API

所有调用都围绕一个核心方法：

```python
response = client.chat.completions.create(
    model="模型名称",
    messages=[...],       # 对话消息列表
    temperature=0.7,      # 随机性（0=确定性，1=更随机）
    max_tokens=1000,      # 最大输出 Token 数
    stream=False          # 是否流式输出
)
```

---

## messages 消息结构

messages 是一个列表，每条消息有 `role` 和 `content`：

```python
messages = [
    # system：角色设定，告诉 AI 它是谁（可选，但推荐加）
    {"role": "system", "content": "你是一个 Python 后端工程师"},

    # user：用户说的话
    {"role": "user", "content": "FastAPI 怎么做参数校验？"},

    # assistant：AI 之前的回答（多轮对话时需要带上）
    {"role": "assistant", "content": "FastAPI 用 Pydantic 做参数校验..."},

    # user：用户的追问
    {"role": "user", "content": "给个代码示例"}
]
```

**三种角色：**

| 角色 | 作用 | 是否必须 |
|------|------|---------|
| system | 设定 AI 的身份、行为、约束 | 可选，但推荐 |
| user | 用户的输入 | 必须 |
| assistant | AI 的历史回答 | 多轮对话时需要 |

**多轮对话的原理**：每次请求都要把之前的对话历史全部带上，AI 本身不记忆。

---

## 返回结果结构

```python
response = client.chat.completions.create(...)

# 获取回答文本
answer = response.choices[0].message.content

# 获取 Token 用量
prompt_tokens = response.usage.prompt_tokens        # 输入用了多少 Token
completion_tokens = response.usage.completion_tokens  # 输出用了多少 Token
total_tokens = response.usage.total_tokens            # 总共多少 Token
```

---

## 关键参数说明

| 参数 | 类型 | 说明 |
|------|------|------|
| model | str | 模型名称，如 "gpt-4o"、"deepseek-chat" |
| messages | list | 对话消息列表 |
| temperature | float | 0-2，越高越随机，越低越确定。写代码建议 0-0.3 |
| max_tokens | int | 限制输出长度，防止回答太长浪费 Token |
| stream | bool | 是否流式输出，True 时逐字返回 |
| top_p | float | 0-1，和 temperature 类似，一般不同时调 |

## 类比理解

如果你熟悉 requests 库，openai SDK 本质上就是这样：

```python
# 用 requests 手动调（不推荐，但帮助理解）
import requests

response = requests.post(
    "https://openrouter.ai/api/v1/chat/completions",
    headers={
        "Authorization": "Bearer sk-or-v1-xxxx",
        "Content-Type": "application/json"
    },
    json={
        "model": "deepseek/deepseek-chat-v3-0324",
        "messages": [{"role": "user", "content": "你好"}]
    }
)
data = response.json()
print(data["choices"][0]["message"]["content"])

# 用 openai SDK（推荐，更简洁）
from openai import OpenAI
client = OpenAI(api_key="sk-or-v1-xxxx", base_url="https://openrouter.ai/api/v1")
response = client.chat.completions.create(
    model="deepseek/deepseek-chat-v3-0324",
    messages=[{"role": "user", "content": "你好"}]
)
print(response.choices[0].message.content)
```

**两种方式结果完全一样，SDK 只是帮你省了拼 URL、设 Header、解析 JSON 的步骤。**

## 流式输出（Stream）

### 什么是流式输出

普通调用：AI 把整个回答全部生成完，一次性返回给你。用户要干等。

流式输出：AI 一边生成一边返回，每生成几个字就发一次，用户能看到文字逐渐出现。

```
普通模式：
用户提问 → [等待 5 秒] → 一次性显示完整回答

流式模式：
用户提问 → [0.3秒] 你 → [0.1秒] 好 → [0.1秒] ， → [0.1秒] 我 → ... 逐字出现
```

**你平时用 ChatGPT、DeepSeek 网页版看到的"打字机效果"，就是流式输出。**

### 为什么需要流式输出

1. **用户体验**：AI 生成一个长回答可能要 5-10 秒，不用流式的话用户要干等，体验很差
2. **做 RAG 时必须用**：知识库问答的回答通常很长，不流式用户会以为卡死了
3. **做 Agent 时必须用**：Agent 执行多步操作，流式能让用户实时看到进度

### 代码对比

```python
# ❌ 普通模式：等全部生成完才返回
response = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "写一篇 500 字的文章"}]
)
# 这里要等好几秒
print(response.choices[0].message.content)  # 一次性全部打印


# ✅ 流式模式：边生成边返回
response = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "写一篇 500 字的文章"}],
    stream=True  # 只加了这一个参数
)
# 立刻开始返回，不用等
for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)
```

### 流式输出的数据结构

普通模式返回的是一个完整的 response 对象。

流式模式返回的是一个迭代器，每次迭代拿到一个 chunk（数据块）：


```python
# 普通模式的返回
response.choices[0].message.content  # "你好，我是AI助手。"（完整文本）

# 流式模式的每个 chunk
chunk.choices[0].delta.content  # "你"（只有几个字）
chunk.choices[0].delta.content  # "好"
chunk.choices[0].delta.content  # "，"
chunk.choices[0].delta.content  # "我是"
chunk.choices[0].delta.content  # "AI助手。"
chunk.choices[0].delta.content  # None（结束）
```

注意区别：普通模式用 `.message.content`，流式模式用 `.delta.content`。

### 在 FastAPI 中使用流式输出（SSE）

后面做项目时，前端要实时显示 AI 回答，就需要用 SSE（Server-Sent Events）：

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

def generate_stream(message: str):
    """生成器函数，逐块返回数据"""
    response = client.chat.completions.create(
        model=MODEL,
        messages=[{"role": "user", "content": message}],
        stream=True
    )
    for chunk in response:
        if chunk.choices[0].delta.content:
            yield f"data: {chunk.choices[0].delta.content}\n\n"
    yield "data: [DONE]\n\n"

@app.get("/chat/stream")
def chat_stream(message: str):
    return StreamingResponse(
        generate_stream(message),
        media_type="text/event-stream"
    )
```

**你有 FastAPI 经验，这个 StreamingResponse + 生成器的模式应该很熟悉。**

### Stream 和 SSE 的区别

这是两个不同的概念，容易混淆：

| 概念 | 全称 | 层面 | 作用 |
|------|------|------|------|
| Stream | Stream（流式输出） | AI 模型层 | 从 AI API 一块一块地拿数据 |
| SSE | Server-Sent Events（服务器推送事件） | HTTP 协议层 | 服务器主动向浏览器推送数据 |

它们的关系：

```
AI 模型（Stream 流式生成）
    ↓ 一块一块地返回数据
你的 FastAPI 后端（接收 stream 数据）
    ↓ 用 SSE 协议转发给前端
浏览器前端（实时显示文字）
```

- Stream 是你从 AI 那里拿数据的方式（`stream=True`）
- SSE 是你把数据推给前端的方式（`text/event-stream`）
- 做一个完整的聊天应用，两个都要用，但它们不是一回事

### 流式输出的注意点

| 注意点 | 说明 |
|--------|------|
| Token 统计 | 流式模式下，Token 用量在最后一个 chunk 里，或者需要自己累计 |
| 错误处理 | 流式传输中途可能断开，需要 try-except 包裹 |
| 拼接完整回答 | 如果需要保存完整回答（比如存数据库），要自己把所有 chunk 拼起来 |

```python
# 拼接完整回答的写法
full_response = ""
for chunk in response:
    if chunk.choices[0].delta.content:
        content = chunk.choices[0].delta.content
        full_response += content  # 拼接
        print(content, end="", flush=True)  # 同时打印

# full_response 就是完整的回答文本
```

## 总结

- `openai` 是一个 Python HTTP 客户端库，不是只能调 OpenAI 的模型
- 几乎所有大模型都兼容 OpenAI 格式，换模型只改 `base_url` 和 `api_key`
- 核心就一个方法：`client.chat.completions.create()`
- messages 有三种角色：system（设定）、user（用户）、assistant（AI 回答）
- 后面学 LangChain、LlamaIndex 时，底层也是调这个 SDK
