# Token 与计费概念

> 理解 Token 和计费机制，直接影响后面做 RAG 和 Agent 时的设计决策。

---

## Token 是什么

Token 是大模型处理文本的最小单位，不是一个字 = 一个 Token。

```
英文：1 个单词 ≈ 1-2 个 Token
中文：1 个汉字 ≈ 1-2 个 Token
代码：符号和关键字各算 Token

示例：
"Hello World" → 2 个 Token
"你好世界" → 4 个 Token（大约）
```

---

## 计费方式

大模型 API 按 Token 数量计费，分输入和输出：

| 计费项 | 说明 |
|--------|------|
| 输入 Token（Prompt） | 你发给模型的内容，包括 System Prompt + 历史对话 + 当前问题 |
| 输出 Token（Completion） | 模型生成的回答 |

通常输出 Token 比输入 Token 贵 2-3 倍。

## 上下文窗口（Context Window）

每个模型能处理的 Token 总量有上限，这就是上下文窗口：

| 模型 | 上下文窗口 |
|------|-----------|
| GPT-4o | 128K Token |
| Claude 3.5 Sonnet | 200K Token |
| DeepSeek V3 | 128K Token |
| Qwen-Plus | 128K Token |

**输入 + 输出的 Token 总和不能超过上下文窗口。**

---

## 为什么这很重要

1. **做 RAG 时**：检索到的文档要塞进 Prompt，文档太长会超出上下文窗口，所以需要分块和筛选
2. **做 Agent 时**：多轮工具调用会累积大量历史记录，需要做记忆管理，不能无限往 Prompt 里塞
3. **成本控制**：一个 RAG 请求可能包含几千 Token 的文档上下文，调用量大时成本会很高
4. **模型选择**：需要处理长文档时，选上下文窗口大的模型

---

## 实用技巧

- 用 `tiktoken` 库（OpenAI 出的）可以在代码里计算 Token 数量
- 开发时打印每次请求的 Token 用量，心里有数
- System Prompt 越短越好，因为每次请求都会带上，累积成本高
